Visión general

Un único repo contiene frontend estático, API Express e infraestructura como código. Vercel sirve ambos (frontend + API) como una función serverless gratuita; los datos viven en una tabla DynamoDB de AWS en modo pay-per-request (coste 0 sin tráfico). GitHub Actions despliega la infraestructura.

Modelo de datos: single-table design

Toda la app usa una sola tabla. El prefijo de pk discrimina la entidad; añadir una entidad nueva no requiere tocar la infraestructura, solo un servicio nuevo.

pkskatributosuso
ITEM<isoDate>#<uuid>id, text, createdAtRegistros de la demo (orden cronológico gratis vía Query)
TU_ENTIDADlo que necesites…Añade el prefijo en src/config/dynamo.js y crea un servicio

Decisiones de diseño

  • Un solo deploy: el frontend lo sirve Express (express.static), así que no hay CORS en producción ni dos pipelines.
  • Fallback sin AWS: sin credenciales la app funciona en memoria. Puedes desplegar en Vercel antes de tener la infra y verlo todo funcionando.
  • Credenciales de mínimo privilegio: el usuario IAM de Vercel solo puede leer/escribir esta tabla; el del pipeline solo tocar este stack.
  • Infra idempotente: CloudFormation con DeletionPolicy: Retain — borrar el stack jamás borra datos.
  • Coste 0 en reposo: Vercel Hobby + DynamoDB on-demand + GitHub Actions gratis en repos públicos.

Pasos, en orden

  1. Probar en local (0 dependencias externas) 
    npm install
npm test        # 6 tests, modo memoria
npm run dev     # http://localhost:3000
  2. Subir a GitHub — crea el repo manualmente si no existe (github.com/new o gh repo create usuario/mi-app --public --source . --push) y haz push. El workflow ci.yml ya pasa en verde sin ningún secreto.

    💡 Puedes marcarlo como Template repository (Settings → General): los repos generados con "Use this template" copian todo, incluidos los workflows de CI, que funcionan igual. Lo único que no se hereda son los secretos: en cada repo generado repite el paso de setup-github-secrets.sh y cambia STACK_NAME/TABLE_NAME en deploy-infra.yml si quieres tabla propia por app.

  3. Desplegar en Vercel (aún sin AWS) — importa el repo en vercel.com/new. Detecta vercel.json y publica. La app funciona ya, en modo memoria.
  4. Crear la infraestructura AWS — dos opciones: 
    # A) Por pipeline (recomendado): sube los secretos y lanza el workflow
./scripts/setup-github-secrets.sh --profile miperfil --repo usuario/repo
gh workflow run deploy-infra.yml

# B) A mano desde tu máquina
aws cloudformation deploy --template-file infra/dynamodb.yml \
  --stack-name vedtemplate-app --profile miperfil --region eu-west-1
  5. Crear el usuario IAM de la app y conectar Vercel 
    ./scripts/setup-aws.sh --profile miperfil
# Pega las 4 variables que imprime en Vercel → Settings → Environment Variables
# y redepliega. La pestaña "Conexiones" pasará a verde.
  6. Sembrar datos de ejemplo (opcional) 
    cp .env.example .env   # pega ahí las mismas credenciales
npm run seed

Despliegue 100% por CLI — flujo real de esta instancia

Esta instancia (vedtemplate.infranettone.com) se desplegó entera desde terminal. Solo hubo dos autorizaciones manuales de un clic: el login por device-code (gh y vercel) y la instalación de la GitHub App de Vercel en la organización para el auto-deploy. 

# 1) Repo público en GitHub, marcado como template
gh repo create infranettone/infranettone-template-vercel-express-dynamodb --public --source . --push
gh api -X PATCH repos/OWNER/REPO -f is_template=true

# 2) Proyecto Vercel + primer deploy (funciona ya, en modo memoria)
vercel link --yes --project <nombre> && vercel deploy --prod --yes

# 3) Hacer la web pública: la Deployment Protection viene activada por defecto
#    PATCH /v9/projects/<id>?teamId=<team>  {"ssoProtection": null}

# 4) Auto-deploy: instalar github.com/apps/vercel en la org y...
vercel git connect --yes

# 5) Infra AWS por pipeline (usuario IAM deploy + secretos + stack + tabla)
./scripts/setup-github-secrets.sh --profile <perfil> --repo OWNER/REPO
gh workflow run deploy-infra.yml

# 6) Credenciales runtime → variables en Vercel por CLI → redeploy
./scripts/setup-aws.sh --profile <perfil>
printf '<valor>' | vercel env add AWS_SECRET_ACCESS_KEY production --sensitive
vercel redeploy <url>

Gotchas ya corregidos en la plantilla: Node 20 no expande globs en node --test; Vercel trunca el dominio *.vercel.app con nombres largos; Vercel define su propia AWS_REGION en runtime; los repos generados desde un template no heredan los secretos de GitHub.

Tests automáticos

npm test ejecuta tests/app.test.js con el runner nativo de Node (node --test, cero dependencias): levanta la app real en un puerto efímero y recorre la API completa en modo memoria — health, status, CRUD entero, validación 400 y 404, y que el frontend se sirve. GitHub Actions (ci.yml) lo ejecuta en cada push y PR.
